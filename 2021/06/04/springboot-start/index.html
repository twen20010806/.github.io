1 Getting Start

1.1 SpringBoot优点

  • 创建独立Spring应用
  • 内嵌web服务器
  • 自动starSter依赖,简化构建配置
  • 自动配置Spring以及第三方功能
  • 提供生态级别的监控、健康检查以及外部化配置
  • 无代码生成、无需编写XML

1.2 SpringBoot缺点

  • 迭代快,需要时刻关注变化
  • 封装太深,内部原理复杂,不容易精通

2 basic concept

2.1 微服务

  • 微服务是一种架构风格

  • 一个应用拆分为一组小型服务

  • 每个服务运行再自己的进程内,也就是可独立部署和升级

  • 服务之间使用轻量级HTTP交互

  • 服务围绕业务功能拆分

  • 可以由全自动部署机制独立部署

  • 去中心化、服务自治、服务可以使用不同的语言、不同的存储技术

2.2 分布式

分布式困难

  • 远程调用
  • 服务发现
  • 负载均衡
  • 服务容错
  • 配置管理
  • 服务监控
  • 链路追踪
  • 日志管理
  • 任务调度
  • …

    分布式的解决

  • SpringBoot+SpringCloud

2.3 云原生

  • 原生应用如何上云

    上云的困难

  • 服务自愈
  • 弹性伸缩
  • 服务隔离
  • 自动化部署
  • 灰度发布
  • 流量治理
  • …

4、自动配置原理

4.1、管理依赖版本号

  • 依赖管理

  • 父依赖的父依赖

  • 自动版本仲裁

  • 可以在pom文件直接修改,依赖就近原则下

  • 也可以,查看spring-boot-dependencies里的key

  • 自动依赖版本管理只会管理版本,不会自动添加依赖

  • 何为starter?

    • spring-boot-starter-*, *为某种场景
    • 例如,spring-boot-starter-web:为web场景的starter
    • starter抽取了常用的依赖并进行了依赖管理
    • 如果非官方的starter
    • *-spring-boot-starter,第三方简化开发的场景启动器
    • 引入非版本仲裁的jar,一定要写版本号
    • 所有场景启动器依赖最底层的依赖是spring-boot-starter

      4.2、自动配置

  • 自动配好Tomcat

    • 引入Tomcat依赖,配置
    • 配置Tomcat

  • 自动配置好SpringMVC

    • 引入SpringMVC全套组件
    • 自动配好SpringMVC常用组件

  • 自动配好Web常见功能,如:字符编码问题

    • SpringBoot帮我们配置好了所有web开发的常见场景

  • 默认的包结构

    • 主程序所在的包及其下面的所有子包里面的组件都会被默认扫描
    • 无需配置以前的包扫描
    • 想要改变扫描路径
      • 使用@SpringBootApplication(ScanBasePackages=”com.atguigu”)
      • @ComponentScan指定扫描路径
    • 一个@SpringBootApplication于三个注解
      @SpringBootConfiguration
      @EnableAutoConfiguration
      @ComponentScan(“com.twen.springboot”)

  • 各种配置拥有默认值

    • 默认配置最终都是映射到一个类上
    • 配置文件的值最终会绑定到某个类上,这个类会在容器中创建对象

  • 按需加载所有自动配置项

    • 依赖按需加载

    • 引入了哪些场景这个场景的自动配置才会开启

    • pringBoot所有的自动配置功能都在

    • 配置类的@Bean注册的组件默认Singelton

      • 以方法名为Bean名
      • Component只能注册简单的组件

    • @Configuration配置类默认也是一个组件

    • 默认@Configuration(proxyBeanMethods = true)

      • 获取到的对象本身就是代理对象,注解中的值为true,获取代理对象
      • 如果为true则不会被代理
      • 代理Bean方法,被代理的@Bean方法会默认单例
      • 外部无论对配置类中@Bean方法配置多少次,默认都是同一个对象

    • 最佳实战

      • 配置类组件之间无依赖管理用lite模式加速容器启动过程、减少判断
      • 配置类组件之间有依赖关系,方法会被调用得到之间的单实例组件、用full模式

  • @Bean、@Component、@Controller、@Service、@Repository

  • @ComponentScan、@Import

    • @ComponentScan扫描包
    • @Import:
      • 将指定类型组件导入,给容器中自动创建出这两个类型的组件
      • 可以写在任意一个组件上
      • 默认组件名字是全类名

  • @Conditional:根据条件装配

    • 加载方法上,当条件成立,该方法才会生效
    • 加载类上
      • 当条件成立,该类下面的所有配置才会生效

  • .xml原生配置文件的引入,@ImportResource(“classpath:*.xml”)

  • 配置绑定:使用javaBean类来进行 配置

    1. @Component+@ConfigurationProperties,注册配置组件

  1. @SpringBootConfiguration

    • 底层@Configuration、代表当前是一个配置类

  2. @ComponentScan

    • 指定扫描哪些，Spring注解

  3. @EnableAutoConfiguration

  1. AutoConfigurationPackage:指定默认包规则

    • 利用Registrar导入一系列组件AutoConfigurationPackages的内部类注册器:registrar

  2. Import(AutoConfigurationImportSelector.class):导入指定包

    • AutoConfigurationImportSelector

    • getAutoConfigurationEntry:调用同类方法得到自动配置入口

      • 同样的类中的方法
      • getCandidateConfigurations:同类中的方法
      • springBoot默认的配置类全在spring-boot-autofiguer/META-INF/spring.factories里

2. 按需开启自动配置项

 * 虽然所有的默认配置类会在启动时默认全部加载,但是会按需配置,但是按照条件装配原则,相应的配置类会被@Conditional所按需装配
 * 所以springboot实际上加载了自动配置类,而自动配置类里又有内部配置类,内部配置类根据条件选择是否配置,以及具体配置所需要的包需要自己导入

3. 修改默认配置

 * 总结
   * springBoot先加载所有的自动配置类,xxxxAutoConfiguration(xxxxProperties.class)
   * 每个自动配置类按照条件进行生效,每一个配置类生效,默认都会绑定配置文件指定的值(),xxxProperties里面拿,xxxProperties和配置文件进行了绑定 
   * 如果生效,生效的配置类给容器中装配很多组件
   * 只要容器中有组件,功能生效
   * 只要用户有自己配置的,以用户优先
   * 定制化配置
     * 用户直接自己@Bean替换底层组件
     * 修改配置文件

4. 最佳实践

 * 引入场景依赖
 * 查看自动配置了哪些
   * 自己分析
   * 在配置文件中debug=true开启自动配置报告 Negative(不生效)\Positive(生效)
 * 是否需要修改
   * 参照文档修改配置项
     * 自己分析,xxxproperties绑定了配置文件哪些比如,mysql
   * 自定义加入或者替换组件
     * @Bean,@component
   * 额外的自定义器

  • 类似于json写法

    • 单行配置需要使用{}
    • 多行配置 key:[space]value
  • 数组,集合
    • 单行配置需要使用[]
    • 多行配置需要使用 -[space]value
  • 自我理解
    • 数组结合的-也许是默认的key
